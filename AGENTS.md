## 3. 协作机制

### 3.1 API 契约协作

**流程：**
1. 后端负责人定义 API Schema（Pydantic）
2. 共同确认 API 文档
3. 后端提供 Mock 接口供前端开发
4. 前端调用 Mock 接口进行开发
5. 后端切换为真实实现

**API 文档管理：**
- 使用 FastAPI 自动生成 Swagger UI
- 文档地址：`/docs` 和 `/redoc`

### 3.2 数据格式约定

**JSON 格式要求：**
- 所有 LLM 返回必须强制 JSON 格式
- 使用 Pydantic 进行校验
- 失败重试1次，再失败降级为模板化结果

**时间格式：**
- 使用 ISO 8601 格式：`YYYY-MM-DDTHH:MM:SSZ`
- 时区统一为 UTC

### 3.3 代码集成流程

```
个人分支开发 → 提交 PR → CI检查 → 代码审核 → 合并到 develop → 部署测试环境 → 合并到 main → 部署生产环境
```

## 5. 决策流程

### 5.1 决策类型

| 决策类型 | 决策方式 | 示例 |
|----------|----------|------|
| **技术选型** | 共同讨论，达成共识 | 选择哪个VLM模型 |
| **设计决策** | 前端负责人主导，后端确认 | UI设计、交互流程 |
| **架构决策** | 后端负责人主导，前端确认 | API设计、数据结构 |
| **紧急问题** | 相关负责人即时沟通 | 线上Bug修复 |

### 5.2 决策记录

- 重要决策必须记录在项目文档中
- 决策理由和权衡需要清晰说明
- 决策变更需要通知所有相关人员

## 6. 冲突解决机制

### 6.1 冲突类型与处理

| 冲突类型 | 处理方式 |
|----------|----------|
| **技术分歧** | 各自阐述理由，数据驱动决策，必要时进行原型验证 |
| **进度冲突** | 优先保障关键路径，协商资源调配 |
| **优先级冲突** | 基于需求重要性和时间约束共同协商 |

### 6.2 升级路径

```
冲突出现 → 双方直接沟通 → 无法达成共识 → 引入第三方协调 → 最终决策记录
```
